### RFC-005: Discrete Event Simulation (DES) Execution Schema

**Status:** Approved for Variant B (MVP)
**Target:** `maritime-port-sandbox`
**Component:** `schemas/` and `api/public/`

#### 1. Context & Motivation
The current `maritime-port-sandbox` provides deterministic state lookups using DCSA standards. For the WinterSim paper, the system must evaluate dynamic routing scenarios generated by the Orchestrator over a simulated timeline. To support the Orchestrator's multi-criteria Pareto analysis, the Sandbox must return standardized, multi-dimensional metrics for each executed scenario, isolating runtime stochastic failures from successful executions.

#### 2. Architectural Decision
We will introduce a new simulation endpoint (e.g., `/api/v1/simulation/execute`) that accepts an array of proposed scenarios and returns a comprehensive `SandboxExecutionOutput`. This schema strictly enforces the collection of Time, Cost, and Risk metrics.

#### 3. Schema Specification (Pydantic v2)
The following models must be implemented in the `schemas/` layer. They serve as the strict contract between the Sandbox (Environment) and the Orchestrator (Agent).

```python
from pydantic import BaseModel, Field
from typing import List, Optional

class ScenarioMetrics(BaseModel):
    # Axis 1: Time and SLA (Minimization)
    total_lead_time_hours: float = Field(..., description="Total transit and processing time")
    average_queue_time_hours: float = Field(..., description="Average vessel waiting time in queue")
    sla_breach_probability: float = Field(..., description="Probability of contract violation (0.0 - 1.0)")

    # Axis 2: Financial Cost (Minimization)
    total_cost_usd: float = Field(..., description="Total logistic costs")
    penalty_cost_usd: float = Field(..., description="Penalties accrued for delays or SLA breaches")

    # Axis 3: Risk and Utilization (Minimization)
    max_yard_utilization_pct: float = Field(..., description="Peak terminal yard utilization (%)")
    bottleneck_severity: float = Field(..., description="Bottleneck severity index (0.0 - 1.0)")

class SimulationEvent(BaseModel):
    timestamp: float
    event_type: str  # e.g., "VESSEL_ARRIVED", "CRANE_FAILED", "QUEUE_OVERFLOW"
    details: str

class ScenarioExecutionResult(BaseModel):
    scenario_id: str = Field(..., description="Identifier matching the Orchestrator's candidate ID")
    execution_status: str = Field(..., description="Allowed: 'SUCCESS', 'INFEASIBLE_DURING_SIMULATION', 'ERROR'")
    metrics: Optional[ScenarioMetrics] = Field(None, description="Populated only if execution_status is SUCCESS")
    critical_events: List[SimulationEvent] = Field(..., description="Timeline of events for explainability")
    failure_reason: Optional[str] = Field(None, description="Solver/DES logs if the scenario failed dynamically")

class SandboxExecutionOutput(BaseModel):
    run_id: str
    simulated_scenarios: List[ScenarioExecutionResult]
```

#### 4. Implications for Existing Architecture
* **Routing:** This does not replace existing DCSA endpoints (which the agent uses to gather context). It acts as an execution sink.
* **Validation:** If a scenario is mathematically impossible to even begin simulating, the Sandbox should return `execution_status: "ERROR"`. If it breaks down during time-stepping due to queues, it returns `INFEASIBLE_DURING_SIMULATION`.
